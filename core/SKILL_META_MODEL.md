### 模式 1: 静默执行协议 (Silent Execution Protocol)

**应用于**: 所有 5 个 Skill

```python
# 伪代码
def execute_skill(input):
    try:
        result = process(input)
        if result.has_warnings:
            result.remarks.append(warnings)  # 不中断
        return generate_report(result)
    except CriticalError as e:
        return f"❌ 终止: {e.message}"  # 仅关键错误才终止
    except Exception as e:
        result.remarks.append(f"处理异常: {e}")
        return generate_report(result)  # 继续执行
```

**你的实战案例**:

```python
def extract_field(text, field_name):
    # 第一层：正则
    result = regex_extract(text, field_name)
    if result != "未提及":
        return result
    
    # 第二层：AI Fallback
    result = ai_extract(text, field_name)
    if result:
        return result
    
    # 第三层：人工兜底
    return "N/A (需人工核查)"
```

### 模式 7: 多层估算模型 🆕

**应用于**: 海外运营商多维解析 Skill（IEPL占比估算）

**核心思想**: 当数据缺失时，使用多层降级策略逐步估算

```
层级1: 直接提取（最优）
   ↓ (失败)
层级2: 关联推算（如：批发业务收入 × 60%）
   ↓ (失败)
层级3: 行业对标（按公司类型取平均值）
   ↓ (失败)
层级4: AI 综合推理（基于多因子评分）
```

**实现代码**:
```python
def estimate_iepl_ratio(carrier_data):
    # 层级1: 直接提取
    if "iepl_ratio" in carrier_data:
        return {"ratio": carrier_data["iepl_ratio"], "confidence": "high (95%)"}
    
    # 层级2: 批发业务推算
    if "wholesale_pct" in carrier_data:
        ratio = carrier_data["wholesale_pct"] * 0.65
        return {"ratio_range": f"{ratio*0.8:.0f}-{ratio*1.2:.0f}%", "confidence": "medium (75%)"}
    
    # 层级3: 行业对标
    benchmarks = {"Tier1": "15-25%", "区域": "5-10%", "移动": "2-5%"}
    company_type = classify_carrier(carrier_data)
    return {"ratio_range": benchmarks[company_type], "confidence": "medium (65%)"}
    
    # 层级4: AI综合推理
    score = calculate_iepl_score(carrier_data)  # 基于海缆、PoP、企业客户
    return {"ratio_range": score_to_ratio(score), "confidence": "low (50%)"}
```

**关键要素**:
- 每层都有明确的触发条件和置信度
- 估算结果必须标注**置信度**和**估算依据**
- 用户可以看到估算是如何得出的

### 模式 10: 基于事实的场景重构 🆕⭐

**应用于**: org-culture-analyst + 所有对话类内容分析

**核心思想**:
> 深度理解对话后，基于事实还原整个对话场景、关系动态和情感氛围，而非机械地逐字提取文本。

**设计哲学**:
- **不是"提取"，而是"重构"**
- **不是"摘录"，而是"还原"**
- **不是"记录"，而是"理解"**

**三大原则**:
1. **忠于事实** - 所有重构必须有原文依据，不添加臆想细节
2. **还原场景** - 重现对话的张力、转折、氛围
3. **保持客观** - 不做价值判断，不臆测动机

**实现框架**:
```
重构的四个维度:
├─ 对话张力 (分歧、转折、共识)
├─ 关系动态 (权力、立场、情感)
├─ 情感氛围 (紧张、协作、急迫)
└─ 决策逻辑 (问题→讨论→决策)
```

**对比示例**:

❌ **简单提取（机械）**:
```markdown
项目经理："我们可以用A方案"
技术负责人："A方案有风险"
领导："我觉得可以试试"
```

✅ **场景重构（智能）**:
```markdown
会议进入方案选择环节。项目经理率先提出了A方案，展现出主动
推进的姿态。但这一提议立即遭到了技术负责人的质疑，后者直接
指出了技术实现的风险。

此时出现了关键的决策节点。领导者介入讨论，用一句"我觉得可以
```

```python
class MetaCognitionEngine:
    """元认知引擎：分析和更新知识系统"""

    def analyze_pattern(self, new_skill):
        """分析新 Skill 中的设计模式"""
        patterns = []

        # 6 大维度分析
        for dimension in self.dimensions:
            result = dimension.analyze(new_skill)
            if result.is_novel:
                patterns.append(result.pattern)

        return patterns

    def calculate_similarity(self, new_pattern, existing_patterns):
        """计算模式相似度"""
        similarities = {}

        for existing in existing_patterns:
            score = (
                self._compare_features(new_pattern, existing) * 0.4 +
                self._compare_scenarios(new_pattern, existing) * 0.3 +
                self._compare_implementation(new_pattern, existing) * 0.3
            )
            similarities[existing.name] = score

        return similarities

    def assess_generalizability(self, pattern):
        """评估模式的通用性"""
        score = 0

        # 场景数量 (40分)
        score += min(len(pattern.scenarios) / 5 * 40, 40)

        # 实现清晰度 (30分)
        if pattern.has_code_snippet:
            score += 30

        # 跨领域适用性 (20分)
        domains = len(set(s.domain for s in pattern.scenarios))
        score += min(domains / 3 * 20, 20)

        # 可追溯性 (10分)
        if pattern.source_skill and pattern.source_code:
            score += 10

        return score

    def update_knowledge_base(self, new_patterns):
        """更新知识库"""
        for pattern in new_patterns:
            # 自动去重
            if self._is_duplicate(pattern):
                continue

            # 添加到知识库
            self.knowledge_base.add(pattern)

            # 更新索引
            self.index.update(pattern)
```

```python
    """

    返回:
        score (int): 通用性评分
        - ≥ 60: 值得添加
        - < 60: 不够通用

    # 1. 适用场景数量 (40分)
    scenario_count = len(pattern.scenarios)
    if scenario_count >= 5:
        score += 40
    elif scenario_count >= 3:
        score += 30
    elif scenario_count >= 2:
        score += 15
    else:
        score += 5

    elif len(domains) >= 2:
        score += 10

    # 4. 可追溯性 (10分)
    if pattern.source_skill and pattern.source_code:
        score += 10

    return score
```

### 模式 12: 浏览器自动化发布 🆕

**应用于**: 小红书内容生成（push_draft.py）

**核心思想**:
> 当目标平台没有开放 API 时，通过 Playwright 浏览器自动化模拟人类操作完成「最后一公里」——从内容生成到平台发布的全自动闭环。关键不在于自动化本身，而在于**踩坑驱动的迭代方法论**。

**适用场景**:
- 目标平台无官方发布 API（小红书、抖音、知乎等）
- 需要将本地生成的内容推送到 Web 平台
- 希望消除手动复制粘贴的重复劳动

**设计原则**:

```
┌─────────────────────────────────────────┐
│        浏览器自动化发布模式              │
├─────────────────────────────────────────┤
│                                         │
│  1. Cookie 持久化登录                    │
│     └─ 首次扫码 → storage_state 保存    │
│     └─ 自动轮询检测登录状态              │
│     └─ 每次操作后更新 Cookie 延长有效期  │
│                                         │
│  2. JS 注入突破平台限制                  │
│     └─ 添加 multiple 属性 → 批量上传    │
│     └─ innerHTML 注入 → 绕过输入限制    │
│     └─ InputEvent 触发 → 欺骗框架更新   │
│                                         │
│  3. DOM 调试闭环                         │
│     └─ 扫描页面按钮 → 打印文字列表      │
│     └─ 关键步骤截图 → 视觉确认          │
│     └─ 选择器失败 → 模糊匹配兜底        │
│                                         │
│  4. 踩坑经验表驱动迭代                   │
│     └─ 每个 bug 记录: 问题|原因|方案     │
│     └─ 写入 SKILL.md 供后续复用         │
│     └─ 避免重复踩坑                     │
│                                         │
└─────────────────────────────────────────┘
```

**实现模板**:
```python
# 浏览器自动化发布核心流程
async def auto_publish(content_dir):
    # 1. Cookie 验证
    if not await check_cookies():
        await login()  # 自动轮询登录状态

    # 2. 打开发布页 + 等待加载
    await page.goto(PUBLISH_URL)
    await page.wait_for_load_state("domcontentloaded")  # 不用 networkidle
    await asyncio.sleep(3)

    # 3. 上传文件（JS 注入 multiple 属性）
    await page.evaluate("document.querySelector('input[type=file]').setAttribute('multiple','')")
    await input.set_input_files(all_files)

    # 4. 填写内容（JS innerHTML 注入 + 事件触发）
    await page.evaluate("""(text) => {
        editor.innerHTML = text;
        editor.dispatchEvent(new InputEvent('input', {bubbles: true}));
    }""", content)

    # 5. 发布（精确匹配按钮文字）
    await page.locator('button:text-is("发布")').click()
    await page.wait_for_url("**/success**")
```

**关键踩坑经验（通用）**:

| 问题类别 | 典型现象 | 通用解法 |
|---------|---------|---------|
| 文件上传 | input 无 `multiple` 属性 | JS 注入属性后批量上传 |
| 富文本输入 | `keyboard.type()` 丢字/截断 | JS `innerHTML` + `InputEvent` |
| 按钮匹配 | `has-text` 匹配到多个元素 | `text-is` 精确匹配 |
| 页面等待 | `networkidle` 超时 | `domcontentloaded` + 固定等待 |
| 按钮找不到 | 平台改了按钮文字 | 扫描所有按钮 → 模糊匹配兜底 |
| 登录检测 | `input()` 在非交互环境阻塞 | 自动轮询 URL/DOM 变化检测 |

### 模式 13: 对话驱动式创作 🆕

**应用于**: 碎片思想出文（thought-to-article）

**核心思想**:
> 当用户的输入不是明确的数据或指令，而是模糊的、未成型的想法时，Skill 不再是"接收输入→生成输出"的单次交互，而是通过多轮对话逐步挖掘、澄清、深化用户的思考，直到素材足够丰富再进入创作阶段。这是从"机器翻译"到"深度采访"的范式转变。

**设计哲学**:
- **共创 > 提取** — 不是"从用户嘴里掏素材"，而是"和用户一起把想法想透"
- **启发 > 引导** — 最好的问题让用户自己说出连自己都没意识到的东西
- **好奇心 > 效率** — 不急着"收集够信息就开写"，真正对用户的想法感兴趣

**三大核心机制**:

```
┌─────────────────────────────────────────┐
│          对话驱动式创作                   │
├─────────────────────────────────────────┤
│                                         │
│  1. 六维完整度雷达                       │
│     └─ 追踪素材丰富程度                  │
│        ├─ 触发（什么事件引发思考）         │
│        ├─ 情感（产生了什么感受）           │
│        ├─ 洞见（意识到了什么）             │
│        ├─ 关联（和更大主题的关系）         │
│        ├─ 故事（有没有画面/场景）          │
│        └─ 转变（想通后有什么不同）         │
│     └─ 4维★★★以上 → 素材充分             │
│                                         │
│  2. 追问技巧库                           │
│     ├─ 深度追问（向下挖）                 │
│     │  ├─ "为什么"追问                   │
│     │  ├─ 具体化追问                     │
│     │  ├─ 情感追问                       │
│     │  └─ 时间线追问                     │
│     ├─ 广度追问（向外扩）                 │
│     │  ├─ 类比追问                       │
│     │  ├─ 反面追问                       │
│     │  ├─ 受众追问                       │
│     │  └─ 假设追问                       │
│     └─ 提炼式追问（向内收）               │
│        ├─ 复述确认                       │
│        ├─ 金句试炼                       │
│        └─ 标题追问                       │
│                                         │
│  3. 自适应对话策略                        │
│     ├─ 模糊方向 → 给具体切口              │
│     ├─ 具体观点 → 追问背后经历            │
│     ├─ 故事经历 → 在关键细节追问          │
│     ├─ 零散碎片 → 帮找共同核心            │
│     ├─ 说不清楚 → 给脚手架选方向          │
│     ├─ 越聊越散 → 收束问题拉回主线        │
│     └─ 已经清楚 → 快速补缺不拉长          │
│                                         │
└─────────────────────────────────────────┘
```

**对比示例**:

❌ **传统模式（单次输入输出）**:
```
用户: "我有个关于AI的想法"
Skill: [直接开始写文章]
结果: 文章空洞，缺乏深度和个人视角
```

✅ **对话驱动模式（多轮共创）**:
```
用户: "我有个关于AI的想法"
Skill: "有意思，是因为最近工作中遇到了什么，还是看到了什么现象让你有感触？"
用户: "我发现很多人学AI的方式是错的..."
Skill: "这个判断挺锐利的。你是看到了什么具体的现象？"
用户: [展开故事和情感]
Skill: [内部追踪六维雷达，判断素材丰富度]
...
Skill: "素材差不多了，让我帮你梳理结构？"
结果: 文章有故事、有洞见、有个人温度
```

**适用场景**（广泛）:
- ✍️ **内容创作** — 碎片想法→文章/演讲稿/博客
- 🎯 **需求澄清** — 模糊意图→清晰需求文档
- 💡 **创意发散** — 粗糙创意→完整方案
- 📋 **深度访谈** — 表面回答→深层洞察
- 🧠 **思维整理** — 混乱思路→结构化表达

**与其他模式的关系**:
- 与"阶段化工作流"（模式 5）互补：Phase 1 是对话阶段，后续 Phase 才进入传统的阶段化执行
- 与"基于事实的场景重构"（模式 10）不同：模式 10 是从已有文本中重构场景，本模式是通过对话从零生成素材
- 与"静默执行协议"（模式 1）形成对比：本模式刻意打破静默，因为对话本身就是核心价值

**为什么值得独立成模式**:
1. ✅ 范式不同 — 现有所有模式都假设输入是明确的，本模式处理的是"输入本身不明确"
2. ✅ 通用性强 — 任何需要深度理解用户模糊意图的场景都可复用
3. ✅ 方法论完整 — 六维雷达 + 追问技巧库 + 自适应策略，构成可复用的对话框架

### 模式 14: 风格DNA提取与迁移 🆕

**应用于**: 碎片思想出文（thought-to-article）

**核心思想**:
> 从参考内容中系统性地提取多维度的"风格DNA"——不是简单模仿某篇文章的语气，而是拆解出开场手法、句式节奏、金句模式、叙事视角、情感基调、段落节奏六个独立维度的风格特征，然后将这组特征作为创作约束条件，迁移到全新内容的生成中。

**设计哲学**:
- **拆解 > 模仿** — 不是"写得像那篇文章"，而是"提取那篇文章的风格基因"
- **维度独立** — 六个维度独立分析，可以混搭组合（如用A的开场手法+B的金句模式）
- **锚点驱动** — 提取完成后生成一句话"风格锚点"，作为后续创作的核心约束

**六维度风格DNA框架**:

```
┌─────────────────────────────────────────┐
│          风格DNA提取框架                  │
├─────────────────────────────────────────┤
│                                         │
│  维度1: 开场手法                         │
│  ├─ 场景画面型（时间+地点+感官描写）      │
│  ├─ 反常识型（建立预期→打破预期）         │
│  ├─ 锐利提问型（一个尖锐的问题开场）      │
│  └─ 金句型（浓缩全文精华的一句话）        │
│                                         │
│  维度2: 句式节奏                         │
│  ├─ 短句密集型（≤15字/句，冲击力强）      │
│  ├─ 长短交替型（长句铺陈+短句打桩）       │
│  └─ 排比递进型（并列或递进结构）          │
│                                         │
│  维度3: 金句模式                         │
│  ├─ 对比转折型 / 重新定义型              │
│  ├─ 具象化抽象型 / 短句三连型            │
│  └─ 问答自设型 / 递进否定型 / 时间对照型  │
│                                         │
│  维度4: 叙事视角                         │
│  ├─ "我"视角（亲历感）                   │
│  ├─ "你"视角（对话感）                   │
│  ├─ "他"视角（客观性）                   │
│  └─ 混合视角（从他→我→你）               │
│                                         │
│  维度5: 情感基调                         │
│  ├─ 冷峻克制 / 热烈真挚                  │
│  └─ 幽默戏谑 / 理性温和                  │
│                                         │
│  维度6: 段落节奏                         │
│  ├─ 段落长度分布                         │
│  ├─ 段间过渡方式                         │
│  └─ 视觉呼吸感（引用块/分隔线）          │
│                                         │
└─────────────────────────────────────────┘
```

**三步提取法**:

```
第一遍: 整体感觉（直觉扫描）
  → "读起来像在听谁说话？读完的情绪是什么？三个形容词？"

第二遍: 结构拆解（看骨架）
  → "开头手法？段落长度？叙事弧线？转折点？"

第三遍: 语言特征（看肌肉纹理）
  → "句子长度？标志句式？用词习惯？金句密度？"

输出: 一句话风格锚点
  → 例："短句密集 + 对比金句 + 个人经历切入宏大议题 + 克制叙事"
```

**对比示例**:

❌ **简单模仿（浅层）**:
```
参考文章语气比较犀利
→ 模仿: 写得犀利一点
→ 结果: 整篇文章语气生硬、用力过猛、缺乏层次
```

✅ **风格DNA迁移（深层）**:
```
参考文章分析:
  - 开场: 场景画面切入
  - 句式: 长短交替，关键处短句打桩
  - 金句: 对比转折型为主
  - 视角: 第一人称，克制但不冷淡
  - 基调: 理性温和，偶尔锐利
  - 段落: 40-80字/段，段间留白
→ 迁移: 按六维度特征分别约束创作
→ 结果: 新文章"气质相似"但不是抄袭，有自己的内容灵魂
```

**风格DNA还可以级联到视觉层**:

```
文风DNA → 文章风格 → 配图风格自动匹配
  科技/洞察 → 几何线条 + 深色背景 + 高亮色
  个人成长 → 有机形状 + 暖色渐变
  商业策略 → 结构化图形 + 克制灰调
  文艺情感 → 留白 + 笔触感 + 低饱和
  犀利评论 → 大胆对比 + 夸张比例
```

**适用场景**（广泛）:
- ✍️ **文章创作** — 模仿某位作者的写作风格
- 📧 **邮件/沟通** — 匹配公司或团队的沟通风格
- 📊 **报告生成** — 延续品牌的文档风格
- 🎨 **视觉设计** — 从文字风格推导视觉风格
- 🤖 **AI Prompt** — 将风格DNA编码为Prompt约束

**与其他模式的关系**:
- 与"模板驱动报告"（模式 9）互补：模板驱动的是结构，风格DNA驱动的是语言质感
- 与"对话驱动式创作"（模式 13）协同：Phase 0 提取风格DNA → Phase 1 对话挖掘 → Phase 3 风格约束下创作

**为什么值得独立成模式**:
1. ✅ 方法论完整 — 六维度框架 + 三步提取法 + 风格锚点，构成可复用的分析体系
2. ✅ 跨领域适用 — 不止写文章，任何"需要匹配某种风格"的内容生成都可用
3. ✅ 可级联扩展 — 风格DNA不仅约束文字，还能级联到视觉、排版等多个维度
