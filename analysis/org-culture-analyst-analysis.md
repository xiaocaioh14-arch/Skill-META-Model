## 🔍 二、逆向工程分析

### 2.1 结构特征

**工作流阶段**: 单阶段（分析 + 生成）

```
输入：会议记录文本
  ↓
【四维文化光谱分析】
  ├─ 01. 认知维度
  ├─ 02. 关系维度
  ├─ 03. 沟通维度
  └─ 04. 行为维度
  ↓
输出：《组织文化 DNA 深度透视》报告
```

**输入输出契约**:
- **输入**: 会议记录文本（通过 `$ARGUMENTS` 传递）
- **输出**: 单一 Markdown 报告，固定结构

**依赖项**:
- ✅ 无 Python 依赖
- ✅ 无 API 依赖
- ✅ 无环境变量
- ✨ **纯 Prompt Engineering** - 完全依靠精心设计的提示词

### 2.2 设计模式识别

#### 🆕 模式 A: 多维度光谱分析框架

**核心特征**:
- 将复杂的定性分析结构化为 4 个维度 × 每维度 2-3 个光谱
- 每个光谱有清晰的两极定义
- 使用 ASCII 进度条可视化位置

**在本 Skill 中的表现**:
```
四维文化光谱:
├─ 认知维度 (3个光谱)
│  ├─ 决策依据: 直觉/权威 <--> 数据/实证 <--> 共识/民主
│  ├─ 时间视界: 短期救火 <--> 中期规划 <--> 长期愿景
│  └─ 错误归因: 个人归罪 <--> 系统归因
│
├─ 关系维度 (3个光谱)
├─ 沟通维度 (3个光谱)
└─ 行为维度 (3个光谱)
```

**与元模型已知模式的关系**:
- ❌ **全新模式** - 元模型中未见类似的"多维度光谱分析"框架
- 部分类似于"加权评分系统"（模式 8），但更侧重定性分析而非定量评分

#### 🆕 模式 B: 证据驱动输出（Evidence-Based Reporting）

**核心特征**:
- 所有结论必须搭配原文引用
- 使用 `> 引用块` 展示文本证据
- "让事实说话"的分析哲学

**关键指令**:
```markdown

```

**与元模型已知模式的关系**:
- 🔄 **已知模式的变体** - 类似"数据质量追踪"（海外运营商 Skill），但应用于定性分析
- 创新点：将"可追溯性"从数据分析扩展到文本分析

#### 🆕 模式 C: 角色人格化（Role Persona）

**核心特征**:

**设计思路**:
- 通过角色设定明确 AI 的立场和风格
- "不寻找病灶，而是还原生态" - 确保中立性
- 防止 AI 做价值判断

## 三、推断的设计思路

### 3.1 核心痛点（从功能反推）

**用户痛点**:
1. **缺少系统化框架** - 分析组织文化时缺少结构化方法，容易流于主观印象
2. **难以量化定性信息** - 文化、氛围等软性因素难以可视化和沟通
3. **容易带入价值判断** - 分析者容易将自己的价值观投射到分析结果中
4. **缺少证据支撑** - 结论缺少原文证据，说服力不足

### 3.2 设计哲学（从实现特征推断）

**识别到的设计原则**:

1. **结构化优先**
   - 体现：四维度、每维度多光谱、固定报告结构
   - 推断：作者相信"好的框架胜过聪明的点评"

2. **中立性至上**
   - 体现：角色设定、反预设指令、"光谱"而非"评分"
   - 推断：作者深知 AI 和人类都容易带入偏见

3. **证据驱动**
   - 体现：所有结论必须搭配原文引用
   - 推断：作者重视可追溯性和客观性

4. **可视化思维**
   - 体现：ASCII 进度条、清晰的视觉层级
   - 推断：作者理解"一图胜千言"，即使是简陋的 ASCII 图

5. **领域专业化**
   - 体现：专注组织文化分析，而非通用文本分析
   - 推断：作者相信"垂直深度胜过水平广度"

### 3.3 技术选型考量（从实现推断）

**为什么选择纯 Prompt Engineering？**

| 选项 | 优势 | 劣势 | 作者选择 |
|------|------|------|---------|
| Python 脚本 + NLP | 可控性强、可量化 | 难以理解语义深度 | ❌ 未选择 |
| API 调用（如情感分析） | 有成熟工具 | 难以捕捉组织文化细节 | ❌ 未选择 |
| 纯 Prompt Engineering | 灵活、语义理解强 | 结果可能不稳定 | ✅ 选择 |

**推断的选型理由**:
- 组织文化分析高度依赖**语义理解**和**上下文推理**
- AI (Claude) 在这方面远胜传统 NLP 工具
- 通过精心设计的提示词，可以弥补结果不稳定的问题

### 3.4 设计权衡（Trade-offs）

| 维度 | 选择 | 放弃 | 推断的考量 |
|------|------|------|-----------|
| **灵活性 vs 一致性** | 一致性（固定模板） | 灵活性 | 确保报告可比较 |
| **深度 vs 广度** | 深度（4维度深挖） | 广度（更多维度） | 避免分析过于分散 |
| **自动化 vs 人工介入** | 自动化（单阶段） | 人工介入 | 提高效率 |
| **量化 vs 定性** | 定性（光谱描述） | 量化（评分） | 保持中立性 |

### 建议 1: ADD - 新增"模式 11: 多维度光谱分析框架"

**目标章节**: `四、关键设计模式`

**更新类型**: 新增章节

**更新内容**:

```markdown
### 模式 11: 多维度光谱分析框架 🆕

**应用于**: org-culture-analyst Skill

**核心思想**: 将复杂的定性分析结构化为多个维度 × 每维度多个光谱

**架构示例**:
```
多维度分析框架
├─ 维度 1
│  ├─ 光谱 A: [极点A] <----> [极点B]
│  ├─ 光谱 B: [极点A] <----> [极点B]
│  └─ 光谱 C: [极点A] <----> [极点B]
├─ 维度 2
└─ ...
```

**设计要点**:
1. **维度选择**: 选择 3-5 个核心维度，避免过多导致分析分散
2. **光谱定义**: 每个光谱的两极必须清晰、对立、无价值判断
3. **可视化**: 使用 ASCII 进度条或简单符号标注位置
4. **证据要求**: 每个判断都需要文本证据支撑

**实现示例**:
```markdown
**决策真理源**
`[权威] ---------|--- [数据]`

深度解读：该组织倾向于数据驱动决策...

> 原文采样："会议中提到'我们需要看一下上季度的转化率数据'..."
```

**适用场景**:
- 定性分析需要结构化（如文化诊断、用户调研）
- 需要在连续光谱上定位而非二元判断
- 需要可视化抽象概念
- 需要保持中立性，避免价值判断

**优势**:
- ✅ 将主观印象转化为结构化洞察
- ✅ 便于横向对比（如对比不同团队的文化）
- ✅ 可视化直观，易于沟通
- ✅ 保持中立性，避免"好坏"二元对立

**注意事项**:
- ⚠️ 维度和光谱的设计需要领域知识
- ⚠️ 光谱两极的定义要避免暗示优劣
- ⚠️ 需要足够的文本证据才能准确定位
```

**实现示例**:
```markdown
## 角色定位

你是一位冷静、客观且洞察力极强的组织观察者。你摒弃"非黑即白"的二元对立，擅长从会议记录的文本流中，提取出一个组织独特的中立文化图谱。你的目标不是寻找"病灶"，而是还原"生态"。

- ✅ 提高输出风格的一致性
- ✅ 明确 AI 的价值观和立场
- ✅ 减少不当输出（如：过度批判、过度乐观）
- ✅ 增强用户对 AI 行为的可预测性

**与"语气风格"规范的区别**:
- 传统方法：简单说"使用专业风格"

**目标章节**: `四、关键设计模式`

**更新类型**: 新增章节（优先级较低）

**更新内容**:

```markdown
### 模式 14: ASCII 可视化 (ASCII Visualization) 🆕

**应用于**: org-culture-analyst Skill

**核心思想**: 使用简单的 ASCII 字符可视化抽象概念，无需外部工具

**实现示例**:
```markdown
**决策真理源**
`[权威] ---------|--- [数据]`

**权力距离感**
`[平等] ---|--------- [森严]`
```

**常见形式**:
- **进度条**: `[====>-----]` 或 `[A] ---|--- [B]`
- **分布图**: `* . * . . * *` (离散点)
- **箭头**: `← 低          高 →`
- **等级**: `★★★☆☆` (5星制)

**设计要点**:
1. **保持简洁**: 控制在 1-2 行内
2. **对齐美观**: 使用等宽字符，保持视觉对齐
3. **标注清晰**: 明确标注两极的含义
4. **避免歧义**: 确保可视化的方向和含义清晰

**适用场景**:
- 需要可视化光谱、进度、分布
- Markdown 环境下无法使用图表工具
- 追求极简风格
- 输出需要纯文本兼容

**优势**:
- ✅ 无需外部工具或库
- ✅ 纯文本，跨平台兼容

### 模式 15: 反预设设计 (Anti-Bias Design) 🆕

**应用于**: org-culture-analyst Skill

**核心思想**: 主动对抗 AI 的固有偏见，防止价值判断泄漏到输出中

**实现示例**:
```markdown
请不要预设好坏，而是在以下四个维度的**光谱**上，精准定位该组织的位置，并寻找文本证据：

（基于以上四个维度，用客观的笔触总结该组织最显著的文化优势与潜在惯性。不给建议，只给镜像。）
```

**常见的 AI 偏见**:
- **西方价值观偏见**: 倾向于认为"民主决策"优于"权威决策"
- **积极偏见**: 倾向于给出正面评价和建议
- **完美主义偏见**: 倾向于指出问题和改进空间
- **二元对立偏见**: 倾向于"好/坏"二分法

**反预设策略**:

1. **显式禁止**:
   ```markdown
   请不要预设好坏
   不给建议，只给镜像
   摒弃"非黑即白"的二元对立
   ```

2. **中性化术语**:
   - ❌ 避免："问题"、"优点"、"缺点"
   - ✅ 使用："特征"、"优势"、"惯性"

3. **光谱代替评分**:
   - ❌ 评分："权力距离: 8/10（过高）"
   - ✅ 光谱："权力距离: [平等] ---|--------- [森严]"

4. **角色设定强化**:
   ```markdown
   你的目标不是寻找"病灶"，而是还原"生态"
   ```

**适用场景**:
- 需要中立、客观的分析（如：文化诊断、用户调研）
- 分析对象涉及价值观和文化差异
- 需要避免"西方中心主义"或其他文化偏见
- 输出会被多元背景的受众阅读

**优势**:
- ✅ 提高分析的中立性和客观性
- ✅ 避免冒犯用户或分析对象
- ✅ 更适合跨文化场景
- ✅ 减少 AI 的"说教倾向"

**局限**:
- ⚠️ 无法完全消除偏见，只能减少
- ⚠️ 某些场景下，价值判断是必要的（如：伦理审查）
```
